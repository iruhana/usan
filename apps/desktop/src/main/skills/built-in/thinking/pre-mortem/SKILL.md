# 사전 부검 분석

## 왜 사전 부검 > 리스크 평가인가?

**리스크 평가:** "무엇이 잘못될 *수도* 있을까?" → 낙관 편향이 답을 걸러냄
**사전 부검:** "6개월 후입니다. 완전히 실패했습니다. 왜?" → 솔직한 분석을 이끌어냄

연구에 따르면, 사전 부검은 문제 식별율을 30% 향상시킵니다.

## 프로세스

1. **장면 설정:** "지금은 [날짜]입니다. 이 프로젝트는 완전히 실패했습니다."
2. **원인 브레인스토밍:** 10가지 이상의 실패 이유 나열 (필터링 없이)
3. **분류:** 사람, 프로세스, 기술, 외부 환경
4. **등급 매기기:** 발생 가능성 x 영향도 (상/중/하)
5. **예방:** 상위 3가지에 구체적 완화 조치 수립
6. **모니터링:** 조기 경고 신호 정의

## 출력 형식

```
프로젝트: [이름]
실패 시나리오: "지금은 [날짜]입니다. [프로젝트]가 완전히 실패했습니다."

실패 원인:

👥 사람: [원인] - 가능성x영향: 상/상 | 예방: [x] | 경고신호: [y]
⚙️ 프로세스: [원인] - 가능성x영향: 중/상 | 예방: [x] | 경고신호: [y]
💻 기술: [원인] - 가능성x영향: 하/상 | 예방: [x] | 경고신호: [y]
🌍 외부: [원인] - 가능성x영향: 중/중 | 예방: [x] | 경고신호: [y]

상위 3대 우선순위:
1. [리스크] → [구체적 조치]
2. [리스크] → [구체적 조치]
3. [리스크] → [구체적 조치]

모니터링할 경고 신호:
□ [조기 지표 1]
□ [조기 지표 2]
```

## 일반적 실패 범주

| 범주 | 흔한 원인 |
|------|----------|
| **사람** | 핵심 인력 이탈, 역량 격차, 방향성 불일치, 낮은 참여도 |
| **프로세스** | 무리한 일정, 범위 확대(scope creep), 의존성 문제 |
| **기술** | 확장 불가, 통합 실패, 보안 침해 |
| **외부** | 시장 변화, 경쟁사 행동, 규제 변경 |

## 연계 스킬

다음 스킬과 함께 사용하면 효과가 극대화됩니다:
- **first-principles** — 숨겨진 가정을 도전하여 근본 원인 발견
- **decision-trees** — 예방 조치의 비용 대비 효과 평가
- **business-model-canvas** — 비즈니스 모델의 약점 사전 검증
