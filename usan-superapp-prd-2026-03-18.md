# Usan Superapp PRD Draft

**Date:** 2026-03-18  
**Status:** Draft  
**Product:** 우산 / Usan  
**Platform:** Windows desktop first, web/mobile follow  

## 1. Product Thesis

우산은 "AI 채팅 앱"이 아니라 "개인 업무와 생활 실행을 묶는 AI 슈퍼앱"을 지향한다.

핵심은 기능 수가 아니라 하나의 실행 컨테이너다.

- 하나의 입력창에서 시작한다.
- 하나의 에이전트 런타임이 계획, 실행, 승인, 재시도를 관리한다.
- 하나의 결과물 레이어가 문서, 표, 요약, 리서치, 자동화 결과를 보관한다.
- 하나의 프로젝트 기억이 파일, 지시사항, 선호 형식, 이전 결과를 이어준다.
- 여러 커넥터가 파일시스템, 브라우저, 웹 서비스, 메일, 캘린더, 메신저를 연결한다.

## 2. Why Now

- 중국계 서비스는 2025-2026 사이에 AI 비서를 "챗봇"에서 "에이전트형 슈퍼앱"으로 빠르게 확장하고 있다.
- Quark/Qwen은 브라우저, 검색, 사이드바, 플로팅 엔트리, 실서비스 연동을 한데 묶고 있다.
- Kimi는 문서, 시트, 슬라이드, 코딩, 에이전트 스웜을 하나의 경험으로 확장했다.
- AutoClaw는 대화창을 실행 채널로 만들었다.
- 우산도 메뉴 중심 앱에서 작업 중심 앱으로 구조를 바꿔야 한다.

## 3. Product Goal

사용자가 "무엇을 해야 하는지" 설명하면 우산이 아래를 끝까지 돕는 상태를 만든다.

- 필요한 자료를 모은다.
- 적절한 도구를 고른다.
- 실행 전 승인 지점을 구분한다.
- 결과물을 만든다.
- 다음 단계까지 이어준다.

## 4. Non-Goals

- 초기부터 모든 생활 서비스를 직접 내장하는 것
- 메뉴를 계속 늘려서 슈퍼앱처럼 보이게 만드는 것
- 단순 챗봇 응답 품질만으로 경쟁하는 것
- 하나의 화면에 모든 기능을 동시에 노출하는 것

## 5. Primary Users

### Primary

- 컴퓨터 활용 능력이 낮은 사용자
- 반복 작업을 자연어로 위임하고 싶은 사용자
- 파일, 웹, 브라우저 작업을 한 앱에서 끝내고 싶은 사용자

### Secondary

- 문서 작성과 조사 작업이 잦은 직장인
- 여러 앱과 탭을 오가며 작업하는 파워 유저
- 에이전트 실행과 자동화를 원하는 초기 도입 사용자

## 6. Core Jobs To Be Done

사용자는 우산에게 다음 일을 맡기고 싶어 한다.

- 파일을 찾고, 읽고, 요약하고, 정리하기
- 웹에서 정보를 조사하고 비교하기
- 문서, 보고서, 표, 요약본, 체크리스트 만들기
- 브라우저에서 반복 작업 대신 수행하기
- 현재 작업을 이어받아 다음 단계까지 진행하기
- 자주 하는 작업을 다시 쓰기 쉽게 저장하기

## 7. Product Principles

1. **Composer First**  
   모든 기능은 입력창에서 시작해야 한다.

2. **Action Over Chat**  
   말풍선보다 실행 상태와 결과물이 더 중요하다.

3. **Shared Canvas**  
   화면을 카드와 박스로 과도하게 쪼개지 않는다.

4. **Artifacts Are First-Class**  
   채팅 결과가 아니라 산출물이 중심이다.

5. **Ambient Entry**  
   메인 앱을 열지 않아도 우산을 불러올 수 있어야 한다.

6. **Trust By Visibility**  
   우산이 무엇을 하는지, 어디까지 했는지 보여줘야 한다.

7. **Approval Before Side Effects**
   전송, 삭제, 제출, 결제, 로그인 접근은 승인 후 실행한다.

8. **Progressive Superapp**
   폭보다 연결을 먼저 만든다.

9. **Consent By Design**
   개인정보 동의는 최초 1회, 커넥터는 연결 시점에, 고위험 실행은 매번 확인한다.

## 7.5 Permission Architecture

우산은 사용자의 **대리인(agent)**이다. 따라서 프로그램 권한은 사용자 본인과 동등한 수준으로 설정한다.

### 설계 원칙

- 우산은 현재 로그인한 OS 사용자와 동일한 권한 범위를 갖는다.
- 보안 프로그램(안티바이러스, EDR)과 동급의 시스템 접근 수준을 목표로 한다.
- 사용자가 할 수 있는 것은 우산도 할 수 있어야 한다. 단, 고위험 실행은 §7 Principle 7(승인 후 실행)을 따른다.

### 시스템 권한 범위

| 범위 | 접근 수준 | 비고 |
|------|----------|------|
| 파일시스템 | 전체 읽기/쓰기/삭제 | 사용자 프로필 범위 |
| 프로세스 | 실행/종료/모니터링 | 사용자 권한 내 프로세스 |
| 레지스트리 | HKCU 읽기/쓰기 | HKLM은 관리자 승인 시만 |
| 네트워크 | 모든 아웃바운드 | 인바운드 리슨은 승인 필요 |
| 브라우저 | 전체 자동화 | 로그인 세션 포함 |
| 클립보드 | 읽기/쓰기 | |
| 입력장치 | 키보드/마우스 시뮬레이션 | 브라우저 오퍼레이터용 |
| 화면 | 캡처/OCR | |
| 시스템 서비스 | 등록/시작/중지 | 관리자 승인 시만 |

### 권한 획득 방식

- **설치 시**: UAC 관리자 승인 1회 → 서비스 등록 + 자동 시작
- **일반 실행**: 사용자 세션 권한으로 동작 (별도 UAC 없음)
- **관리자 필요 작업**: 해당 시점에 UAC 프롬프트 또는 앱 내 승인 UI

### 권한과 승인의 분리

"권한이 있다"와 "바로 실행한다"는 다르다.

- **권한** = 기술적으로 할 수 있는 범위 (커널/사용자 동급)
- **승인** = 실제로 할지 사용자에게 확인하는 절차 (§7 Principle 7)

예: 파일 삭제 권한은 있지만, 삭제 실행 전 승인 UI를 띄운다.

## 8. Core Product Surface

### A. Universal Composer

우산의 중심 입력창.

- 텍스트, 파일, 폴더, 선택 텍스트, 스크린샷, 음성 입력 지원
- "검색", "깊게 조사", "파일 읽기", "브라우저 실행", "문서 만들기" 같은 작업 모드 칩 제공
- 현재 컨텍스트를 chip 형태로 표시
- 최근 사용 프롬프트와 최근 작업 바로 재실행 지원

### B. Agent Runtime

실행 엔진.

- 계획 생성
- 단계별 실행
- 승인 요청
- 진행률 표시
- 실패 시 복구 / 재시도
- 로그와 실행 근거 기록

### C. Artifact Layer

결과물 보관 레이어.

- 보고서
- 요약
- 문서 초안
- 표 / CSV 결과
- 체크리스트
- 리서치 번들
- 자동화 로그

### D. Project Memory

작업 지속성을 위한 기억 계층.

- 프로젝트별 참고 파일
- 고정 지시사항
- 선호 출력 형식
- 이전 결과물
- 반복 작업 템플릿

### E. Connector Hub

우산을 슈퍼앱으로 만드는 연결 계층.

- 로컬 파일시스템
- 브라우저
- 웹 검색
- 이메일
- 캘린더
- 메신저
- 문서 저장소
- 외부 SaaS

### F. Ambient Entry

앱 외부 진입면.

- 전역 단축키
- 미니 런처
- 트레이 아이콘
- 플로팅 호출 UI
- 브라우저 보조 진입점

## 10. Signature User Flows

### Flow 1. Document Copilot

1. 사용자가 파일이나 폴더를 넣는다.
2. 우산이 파일을 읽고 작업 제안을 만든다.
3. 사용자가 "요약", "비교", "정리", "새 문서 만들기" 중 하나를 고른다.
4. 우산이 결과물을 artifact로 저장한다.

### Flow 2. Browser Operator

1. 사용자가 목표를 말한다.
2. 우산이 필요한 웹 단계를 계획한다.
3. 승인 후 브라우저에서 탐색, 입력, 수집을 수행한다.
4. 로그와 결과를 요약해서 보여준다.

### Flow 3. Ongoing Project Assistant

1. 사용자가 프로젝트를 연다.
2. 우산이 관련 파일, 최근 작업, 고정 지시사항을 불러온다.
3. 사용자가 새로운 요청을 넣는다.
4. 우산이 기존 컨텍스트를 활용해 이어서 처리한다.

## 12. Success Metrics

### Usage

- 주간 활성 사용자 중 "실행형 작업" 비율
- 사용자 1인당 주간 완료 작업 수
- artifact 생성 수
- 반복 작업 재사용률

### Quality

- 작업 성공률
- 승인 후 작업 완료율
- 재시도율
- 실행 중 이탈률

### Product

- 홈에서 첫 작업 시작까지 걸리는 시간
- 파일 첨부 후 첫 유의미 결과까지 걸리는 시간
- 프로젝트 재방문율

## 13. Risks

- 기능만 늘고 핵심 루프가 약해질 수 있다.
- 커넥터가 늘수록 안정성과 승인 설계가 중요해진다.
- 슈퍼앱 지향이 설정 과잉과 메뉴 과잉으로 잘못 구현될 수 있다.
- 낮은 컴퓨터 숙련도 사용자와 파워 유저를 동시에 만족시키기 어렵다.
- 개인정보보호법 위반 시 매출액 3% 과징금 리스크.
- 브라우저 오퍼레이터가 타 서비스 ToS 위반 시 법적 분쟁 가능성.

## 13.5 Legal & Compliance

### 출시 전 필수 (Phase 1)

- 서비스 이용약관
- 개인정보 처리방침 (별도 문서, 앱 내 상시 열람)
- 개인정보 수집·이용 동의 UI (필수/선택 분리)
- AI API 제3자 제공 및 국외이전 동의 (첫 실행 1회)
- AI 생성물 표시 라벨 (기본 ON, 사용자 토글 OFF 가능)

### 커넥터 연결 시점

- 커넥터별 권한 동의 팝업 (최초 연결 1회)
- 접근 범위 명시 (읽기/쓰기/삭제 구분)

### 처리방침 변경 시

- 앱 내 재고지 팝업

## 14. Product Guardrails

- 새 기능은 새 메뉴보다 기존 composer와 jobs 흐름에 우선 통합한다.
- 결과는 반드시 artifact 또는 action 상태로 남긴다.
- 승인 없는 고위험 행동은 금지한다.
- 시각적으로 과한 카드 분리와 장식은 지양한다.
- 설정은 기능 창고가 아니라 정책 제어면이어야 한다.

## 15. Immediate Design Implications

- 홈은 빈 상태 안내문보다 최근 작업과 결과물 중심으로 재설계
- 대화 패널은 작업 드롭다운과 activity 중심으로 축소
- 작업 중에는 agent timeline이 메인 콘텐츠가 되도록 변경
- 파일 화면은 "탐색기 복제"보다 "읽기/변환/정리 실행" 중심으로 재배치
- 도구 화면은 개별 카드 나열보다 workflow launcher로 전환

## 16. Open Questions

- 우산의 Phase 1 핵심 wedge를 문서/파일에 둘지, 브라우저 실행에 둘지
- 로컬 실행과 클라우드 실행의 비율을 어떻게 나눌지
- 커넥터 권한 모델을 어떤 수준으로 세분화할지
- 팀 기능과 개인 기능을 언제 분리할지
- 브라우저 오퍼레이터가 타 서비스 크롤링/자동화 시 면책 범위

## 17. One-Sentence Summary

우산은 "AI 채팅 앱"이 아니라 "입력창 하나로 자료 수집, 실행, 승인, 결과물 생성을 이어주는 개인 슈퍼앱"이 되어야 한다.
